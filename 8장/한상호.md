# 🔖 Ch8. 품질과 실용주의 사이에서 균형을 유지하라

> 작성 일자 : 2025.05.03 / 작성자 : 한상호

## 💫 기억에 남는 문구

- `p.214` : 올바른 추상화를 선택하는 것이 얼마나 중요한지 설명한다. "올바른 추상화를 선택하면 프로그래밍이 설계부터 자연스럽게 흘러간다."
- `p.217` : 자신이 해결하려는 일반적인 문제를 제대로 이해하기 전에 추상화를 너무 일찍 만들면 특정 유스케이스에 과하게 맞춘 설계가 탄생할 수 있다.
- `p.218` : 단순한 것은 하나의 역할을 맡고, 하나의 작업을 수행하고, 하나의 목표를 달성하고, 하나의 개념을 다룬다고 설명한다.
- `p.219` : 좋은 추상화를 설계하려면 수고가 필요하다. 다른 사람의 추상화를 연구해서 좋은 추상화를 만드는 방법을 배워라.
- `p.222` : 작성 중인 코드가 실제 프로덕션에 배포될지 확실하지 않을 수도 있다. 배포가 불투명한 제품의 테스트를 작성하겠다는 동기를 느끼기는 쉽지 않다.
- `p.226` : 초기에는 항상 문제를 완전하게 이해하지 못하므로 약간의 부채가 발생하는 것은 불가피하다. 소프트웨어를 만드는 자연스러운 과정일 뿐이다.

## 💡 느낀 점

1. 이전에는 문제 해결이 중점이 아닌, 새로운 기술 도입에 중점을 두고 개발을 했었다. 그러다 보니 사용하는 기술은 늘어났음에도 실질적으로 효과가 크지는 않았다. 이는 오히려 복잡성을 증가시키며 때로는 오버엔지니어링이 되기도 했다.
2. 그렇기에 요즘에는 오버엔지니어링에 대해서 경각심을 가지면서, `이 기술이 정말로 문제해결에 필요한가?`를 먼저 고민하고 있다. 그렇기 때문에 구현보다 설계에 훨씬 많은 시간을 쓰게 되는 듯하다.
3. 개발을 하다 보면 코드가 마음에 들지 않음에도, 마감시간 때문에 배포를 하는 경우가 종종 있다. 이럴 때에는 마음이 불편하지만, 시간은 한정적이기에 적절한 트레이드오프가 필요한 부분인 것 같다.
4. 더욱 중요한 것은 이후에, 책임감을 가지고 기술 부채를 상환하며 & 코드를 지속적으로 개선하려는 태도를 가지는 것이라고 생각한다.
